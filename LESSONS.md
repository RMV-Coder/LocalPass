# LESSONS.md — decisions, conventions, and lessons learned

Living log for the LocalPass build. [PRD.md](PRD.md) is the *what*; this file records *how* we're building it and what we learned along the way. Read this before starting any new work unit.

## Conventions

- **Commits:** conventional commits, small atomic diffs, one reviewable unit each.
- **Workspace:** crates live under `crates/*` (globbed in the root `Cargo.toml`, so adding a crate never touches the root manifest).
- **Delegation:** major implementation tasks go to Opus subagents with scoped prompts and explicit acceptance criteria. The orchestrator reviews output, runs `cargo fmt --check`, `cargo clippy -- -D warnings`, and `cargo test`, then commits. Subagents never run git commands.
- **Crypto boundary:** `lp-crypto` is the only crate allowed to depend on crypto primitive crates. Everything else consumes its misuse-resistant high-level API.
- **Fixed cross-crate contracts** (mirrored in `docs/specs/`):
  - Envelope v1: `0x01 || nonce(24 bytes) || ciphertext+tag`, XChaCha20-Poly1305, AAD carried out-of-band.
  - HKDF label namespace: `localpass/v1/<purpose>` (labels mandatory, never empty).
  - Key hierarchy names: MUK → AccountKey → VaultKey → ItemKey; IndexKey derived from VaultKey with label `localpass/v1/index`.

## Decisions

- **2026-07-04 — PRD v1.0 ratified** including the 8 decision-log items (PRD §11): server-enforced token scopes (P2), persisted incremental encrypted search index, MPL-2.0 for GUI, `localpass://` with `op://` alias, best-effort passkeys (macOS/iOS → Windows → Linux), file-based sync as onboarding default, pairing-code-gated relay enrollment, keep-forever retention + visible stats + prune tooling.
- **2026-07-04 — Toolchain:** Rust stable (1.96.1 at time of setup), edition 2024, resolver 2. `rust-version = 1.90` floor.
- **2026-07-04 — Supply chain:** cargo-deny gates in CI from day one (advisories, license allowlist, unknown sources).
- **2026-07-04 — On-disk layout:** one **account store** SQLite file (KDF params, wrapped AccountKey, device keys, vault registry, settings) plus **one SQLite file per vault**. Rationale: per-vault portability and sync, blast-radius isolation, matches PRD "single-file vaults". Specified in `docs/specs/vault-format.md`.
- **2026-07-04 — Durability:** uniform `PRAGMA synchronous=FULL` (WAL) on the account store AND vault files. NORMAL could lose the last committed write on power loss — unacceptable for a just-saved credential; human-scale write rates make the fsync cost irrelevant.
- **2026-07-04 — Plaintext minimization:** `item_type`, `favorite`, `folder_id` live INSIDE the encrypted item payload, not as plaintext columns (spec-review tightening). PRD §6.3 fixes plaintext to ids/counters/timestamps; type distribution is targeting info. Type/folder/favorite filters run via the encrypted index.
- **2026-07-04 — Sync MVP boundary confirmed:** single-user multi-device only; team membership-change ops are a named P2 extension point (new signed op_kind gated on admin keys), not designed now.
- **2026-07-04 — Index prefix matching at query time**, not stored prefix tokens (BTreeMap range scan over posting keys). Stored prefixes bloated segments ~4×; query-time scan gives identical results at far lower write amplification. Spec §2/§6 semantics preserved.
- **2026-07-04 — Index segment tuning knobs** (`LP_INDEX_SPLIT`/`TARGET`/`MERGE` env vars) exist for tests only; production uses spec constants (256/512/256). Tuning is explicitly non-format-fixed per spec §3.
- **2026-07-04 — Typed key transport across devices:** moving a key between devices goes through `lp_crypto::seal_key_for` / `SealingKeyPair::open_key` (SymmetricKey in, SymmetricKey out) — never a raw-bytes accessor. Share-blob AADs bind vault id + recipient device id. Pattern: when a layer above needs "the impossible" from lp-crypto, add a narrow typed primitive to lp-crypto rather than widening byte access.
- **2026-07-04 — Sync causality is now exact (op wire v2).** Replaced scalar-Lamport concurrency with a per-op observed-heads version vector (`observed`: device→highest-applied-seq at author time), signed + hash-chained. happens-before: same-device = seq order; cross-device = `b.observed[a.device] >= a.seq`; concurrent = neither. Lamport is retained ONLY as the LWW total-order tiebreak (who wins among concurrent writers) — cleanly split from concurrency detection (whether they're concurrent). The two-mechanism split is the key idea: a scalar clock can order but cannot detect concurrency (a→b ⟹ a.lamport<b.lamport, never the converse). Fixed the concurrent-delete-vs-edit case (#17 resolved).
- **2026-07-04 — Foreign-format crypto boundary:** lp-crypto stays the sole home of LocalPass's OWN vault crypto; `lp-porter` may use the `age` crate for foreign-format archive crypto (it resolves to the exact same primitive crate versions — no duplicated crypto surface). KDBX import stubbed for this reason: the `keepass` crate pulls a parallel crypto stack (~85 transitive crates); revisit as a feature-gated opt-in.
- **2026-07-04 — Standalone `age` CLI cannot decrypt non-interactively** (passphrase input is TTY-only by design; winpty can't fake a console headless). Exit-hatch compatibility is proven by construction: exact `age-encryption.org/v1` magic + standard scrypt stanza + official-crate provenance + in-crate round-trip. A one-time manual `age -d backup.age` check by a human is still worth doing before 1.0.
- **2026-07-04 — CLI secret-key storage:** `<profile>/secret-key` file (0600 on Unix) is the MVP stand-in for OS-keychain storage (keychain is P2). Documented in `--help` and the Emergency Kit output.
- **2026-07-04 — Browser native-messaging host (`lp-native-host`, new AGPL crate):** `localpass-native-host` is a daemon CLIENT (like the CLI) speaking Chrome/Firefox native messaging on stdio. Framing is a **native-endian** u32 length prefix + UTF-8 JSON with a **1 MiB inbound cap** (Chrome's limit) — distinct from the daemon IPC's little-endian frame; use `to_ne_bytes`/`from_ne_bytes` to honor the spec verbatim. The host holds NO keys and issues ONLY three daemon requests — `Status`/`MatchLogins`/`FillLogin` (fill-scoped by construction; it cannot ask for a raw item read, export, or any mutation). `credentials_for` never returns a password; only `fill` does, for one user-selected item.
- **2026-07-04 — Autofill origin match is server-side + defense-in-depth:** registrable-domain (eTLD+1) matching lives in `lp_daemon::origin` (one authoritative impl); the host pre-filters candidates with it AND the daemon **re-validates** the item's URL against the origin on `FillLogin` before releasing the secret — a mismatch returns an error, never the password (PRD §8 T7). Lookalikes (`evil-example.com`, `example.com.evil.com`) never match; bare public suffixes (`com`, `co.uk`) and IP/`localhost` have no registrable domain and are refused.
- **2026-07-04 — Pragmatic eTLD+1 without a PSL crate (documented MVP limit):** `lp_daemon::origin` uses a conservative heuristic (last two labels, or last three when the last two are in a small built-in `MULTI_PART_SUFFIXES` set like `co.uk`/`com.au`/`co.jp`) rather than pulling `psl` (bakes the whole ~9k-entry list into the binary) or `publicsuffix` (needs the list shipped). The failure mode is over-matching *within* a shared registry suffix, never across unrelated domains — the core phishing protection holds. Full PSL is a follow-up; only `registrable_domain`'s internals change when it lands, not the wire protocol.
- **2026-07-04 — Native-host profile alignment:** the daemon serves one profile and the host doesn't know it a priori. The host sends a best-guess `profile` and, on `WrongProfile { expected }`, adopts it and retries once — so the host stays aligned with whatever profile the running daemon serves without needing to resolve the platform data dir (avoids a `directories` dep). Locked/unreachable daemon → immediate `locked`/`unavailable` reply, never a hang (PRD §4.7 "never blocks the browser").
- **2026-07-04 — `browser register`/`unregister`:** writes the native-messaging manifest (`com.localpass.host`, path to `localpass-native-host`, `type:stdio`, `allowed_origins` for Chrome / `allowed_extensions` for Firefox) to the per-OS location; on Windows also the `HKCU\Software\<vendor>\NativeMessagingHosts\com.localpass.host` default value pointing at the manifest (via a tiny `windows-sys` `Win32_System_Registry` helper — same crate family lp-daemon uses, no new surface). No published extension id yet → a documented placeholder, overridable with `--extension-id`. Host binary resolved as a sibling of `localpass` (installer lays them side by side), overridable with `--host-path`.

## Lessons learned

- **2026-07-05 — Encrypted file attachments (Wave 15a+15b).** Files attach to an item: each becomes a content-addressed encrypted blob at `<profile>/attachments/<vault_id>/<blake3(ciphertext)>.blob` (never in SQLite), sealed under a fresh per-attachment key wrapped by the item's ItemKey; filename sealed under the ItemKey too. AADs bind vault+attachment id (relocation fails closed). 50 MiB cap. The daemon surface is **path-based**: AddAttachment reads a source_path, GetAttachment writes a dest_path (force to overwrite) — blob bytes never cross the IPC pipe and never enter the webview (a stronger boundary than reveal_field). Native file/save pickers via the tauri dialog plugin. **Limitation:** attachments are LOCAL-ONLY — no op authored, not synced yet (follow-up: ship blobs through the file channel). **How to apply:** for bulk/binary secret payloads, keep them out of the DB and off the pipe — content-address the ciphertext on disk and move data daemon↔disk by path.
- **2026-07-05 — Subagents sometimes skip a final `cargo fmt`.** Two attachment waves left an unformatted new test file / command fn that failed `cargo fmt --all --check` on the orchestrator's independent run. **How to apply:** always run `cargo fmt --all` yourself before committing a subagent delivery, even when the agent reports fmt clean — a file edited after its last fmt pass slips through.

- **2026-07-05 — Never put backticks in a `git commit -m` message via the Bash tool.** The shell interprets `` `...` `` as command substitution and silently eats the line (a `localpass run …` line vanished from a commit message). Use `git commit -F <file>` (write the message with Write first) for any message containing backticks, `$()`, or `$VAR`. Amend from a file to fix.
- **2026-07-05 — .env developer UX is GUI-only because env entries are already fields.** env_set entries surface as per-KEY fields, so "export as .env" reuses `reveal_field` per entry (each disclosure stays gesture-driven + audit-logged — no bulk-dump command) and paste-import just builds the existing NewItemInput entries. The `localpass run --env-set` injection engine was built in Wave 4a; the GUI only had to surface a copyable command. Pattern: check whether an existing gesture-scoped primitive composes before adding a new bulk secret path.

- **2026-07-05 — A stray `localpass daemon start` breaks the ssh_agent test.** The SSH agent binds the machine-global `\\.\pipe\openssh-ssh-agent` (only one per machine); the `ssh_agent` integration test fails ("wrong key count") if any localpass daemon — e.g. one left running from manual testing — already holds that pipe, because the test connects to the stray daemon. It passes in isolation. **How to apply:** before running the full `-p lp-daemon` suite, `taskkill //IM localpass-daemon.exe`; and treat a lone ssh_agent failure amid green device_sync/create_* as environment contamination, not a code regression. (Follow-up: the ssh_agent test should isolate or skip when the fixed pipe is taken.)
- **2026-07-05 — Device linking in the GUI (Wave 13):** additive daemon requests expose the already-built lp-sync engine (ExportIdentity/TrustDevice/ListPeers/SyncSetup/Push/Pull/Status/Adopt/ShareVaultToDevice). Trust enforces fingerprint confirmation **server-side** (`sync::trust_device` refuses an empty or mismatched `expected_fingerprint`) — the GUI checkbox is only a usability aid. Identity strings + fingerprints are public (public keys + hash); no secret crosses these commands. Proven by a two-daemon share→adopt→bidirectional-sync integration test.

- **2026-07-05 — The GUI is now standalone (onboarding + item CRUD).** Account creation goes through a new additive daemon `CreateAccount` request (the GUI stays a client and never holds keys) that writes `<profile>/secret-key` byte-for-byte as the CLI's init so the unlock path reads it; `Response::AccountCreated` carries the Secret Key once over the same-user pipe for the Emergency Kit, and `Response`'s Debug is kind-only so `--verbose` never logs it (tested). Item edit uses `Option<String>` for secret fields (`None` = keep current) so an un-revealed password is never wiped — the correct pattern for editing masked data. **How to apply:** when a GUI needs an action the daemon didn't expose, add an additive daemon request rather than giving the client crate direct vault access; and route "no account yet" as a distinct UI state (don't conflate it with "locked").

- **2026-07-05 — GUI must auto-start the daemon; the desktop bundle ships it as a sidecar.** The GUI is a daemon client with no direct-unlock fallback, so a first-run user hit a misleading "Unlock failed." with no daemon running. Fix: `daemon::ensure_running()` spawns the daemon via the CLI's `spawn_detached` on GUI launch (`ensure_service` command) and before unlock. For a packaged install, `tauri.conf.json` `bundle.externalBin: ["binaries/localpass-daemon"]` bundles it — **verified** (via `msiexec /a` extract) that Tauri strips the target triple and co-locates `localpass-daemon.exe` beside `localpass-desktop.exe`, which is what `DaemonExe::Auto`'s sibling lookup needs. `npm run bundle` (scripts/prepare-sidecar.mjs → tauri build) is the self-contained-installer command. **How to apply:** any Tauri app that shells to a helper binary should bundle it as an externalBin sidecar and confirm the installed layout with an admin/extract install, not assume the on-disk name.

- **2026-07-04 — Run the real supply-chain gate before declaring "clean."** The manual license audit during the porter wave concluded cargo-deny "excludes workspace members" — it does not by default. Running actual `cargo deny check` surfaced three real config gaps + two genuine advisories. Fixes: (1) mark workspace crates `publish = false` so they're treated as private (this makes both `licenses.private.ignore` AND `bans.allow-wildcard-paths` apply — path deps are only allowed in unpublished crates); (2) document-ignore RUSTSEC-2023-0071 (`rsa` Marvin) with the rationale that we only *sign* locally with RSA SSH keys (no decryption, no network-observable timing); (3) document-ignore RUSTSEC-2026-0173 (`proc-macro-error2` unmaintained, a build-time transitive of `age`). **How to apply:** actually run the gate locally; don't infer its result. Ignores must carry an inline rationale and be mirrored in SECURITY.md — never a bare id.

- **2026-07-04 — Adding a write to `unlock` can corrupt read-only flows.** The audit log made every unlock append a record, which corrupted `backup verify` (it unlocks the backup copy, mutating it and tripping the manifest hash check). Fixed with an `unlock_quiet` path for verify/restore. **How to apply:** when you add a side effect to a hot, widely-called primitive (unlock, open, get), grep every caller for read-only/immutability assumptions before shipping — a "verify must not mutate" invariant is easy to break silently.
- **2026-07-04 — Audit log is plaintext metadata by design** (ids/kinds/timestamps/field-names, never secret values or vault/item names), in an `audit_log` table in the account store, device-local and never synced. Integrity is the BLAKE3 chain, not confidentiality — the point is a log that stays inspectable while the vault is locked and answers "who read what, when." Single-logged by the Session-holding side (daemon in proxied mode, CLI in direct mode) so proxied+direct never double-count. Verified the privacy property by dumping the table after a real reveal.

- **2026-07-04 — GUI is a detached workspace under `apps/`.** Root `Cargo.toml` has `exclude = ["apps"]` and `apps/desktop/src-tauri/Cargo.toml` has its own empty `[workspace]`, so the AGPL core's `cargo test --workspace`/clippy/doc stay fast and NEVER build the MPL-2.0 Tauri app. Acceptance gate for GUI work: confirm from repo root the core gates pass AND show no `Compiling localpass-desktop`/`tauri`. The GUI is a daemon client (like CLI + native-host) — it holds no keys.
- **2026-07-04 — Enforce the secret boundary structurally, not by convention.** The GUI's `model::item_view_masked` is one `#[must_use]` choke point that drops secret field values, with a test asserting the *serialized JSON* never contains a planted secret even if the daemon leaked it. Pattern for any render layer: make the masking a single pure function and test it at the serialization boundary, so "secrets don't reach the view" is a property the type system + one test guarantee rather than something every call site must remember.
- **2026-07-04 — `.claude/` is local tooling config, gitignored** (launch.json, scheduled_tasks.lock) — never commit it.

- **2026-07-04 — Verify crypto outputs against an independent consumer, not just round-trips.** ssh-key 0.6.7 has a real RSA bug (CRT primes passed as `p,p` instead of `p,q`) that a self-round-trip test can mask; it surfaced because signatures were verified against the public key and then against real `ssh-add`. **How to apply:** for any signing/encryption path involving a third-party format, include at least one test that validates output with an independent implementation (another crate's verifier, a system tool), not only our own decoder.

- **2026-07-04 — Interrupted subagents leave recoverable work.** A machine restart killed the Wave 6 agent mid-flight, but its working-tree output survived (~1,900 lines, compiling, 6/7 tests passing). Recovery cost was one bad test assumption (a CLI-layer `secret-key` file referenced from an lp-vault-layer test), a missing `cargo fmt`, and absent CLI-level tests. **How to apply:** after any interrupted agent, run the gates on the working tree first — finishing partial work is usually far cheaper than relaunching from zero; look specifically for layer-confusion mistakes near where the work stopped.
- **2026-07-04 — Timing-window tests need process-spawn headroom.** The daemon autolock test used a 1s idle window; post-reboot process spawn ate it and the test flaked. Windows CLI process spawn + connect can cost ~1s cold. **How to apply:** in tests where a fresh CLI process must land inside a timing window, size the window ≥4s.

- **2026-07-04 — Daemon transport & access control (decision):** Windows named pipe `\\.\pipe\localpass-<user>` with a DACL granting only the current user's SID (`D:(A;;GA;;;<sid>)`) + `PIPE_REJECT_REMOTE_CLIENTS` + `FILE_FLAG_FIRST_PIPE_INSTANCE`; Unix UDS 0700/0600 + `SO_PEERCRED` euid check. The OS-enforced same-user restriction is the authentication (PRD §8 T8); the master password crosses the channel in clear only because of it. Detached spawn MUST use `CreateProcessW` with `bInheritHandles=FALSE` — `std::process::Command` inherits the launcher's stdout pipe and deadlocks a piped `daemon start`. Threaded std, no tokio; one mutex serializes the `!Sync` Session; request read/write happens outside the lock so a hung client can't block auto-lock.
- **2026-07-04 — PowerShell `'text' | exe --password-stdin` is unreliable for password piping** (5.1 encoding/newline quirks made a correct password read as wrong). The env-var and interactive paths are fine. **How to apply:** when smoke-testing stdin secret input on Windows, prefer `$env:LOCALPASS_PASSWORD` or a real pipe from a file; don't trust a PowerShell string-literal pipe, and don't treat its failure as a code bug without cross-checking another input path.
- **2026-07-04 — Smoke-test Windows CLIs from PowerShell, not git-bash.** MSYS argument conversion turns a bare `/c` into `C:\`, so `localpass run -- cmd /c ...` appeared broken from bash (interactive cmd banner, wrong exit codes) while the binary was correct. **How to apply:** for Windows child-process behavior, verify via the PowerShell tool; treat git-bash smoke failures involving `/`-prefixed args as suspect before blaming the code.

- **2026-07-04 — Gate commits on `cargo test`'s exit code, never a grep'd pipeline.** A `cargo test | grep "test result" && git commit` chain committed a broken test because grep's exit code masked the failure (caught and amended immediately). **How to apply:** run tests as their own command, branch on `$?`, and only then commit.
- **2026-07-04 — AAD component encoding (fixed contract):** AAD strings are UTF-8, joined with a single `|`: labels verbatim, ids as 32-char lowercase hex (no hyphens), integers as decimal ASCII. Chain-genesis hash input is raw-byte framed: `label_utf8 || vault_id(16) || device_id(16)`.
- **2026-07-04 — Review subagent crypto with a hardening checklist, not just tests.** The lp-crypto delivery was excellent (constructions, hygiene, oracle-resistance all correct), yet still missed X25519 low-order-point rejection (`SharedSecret::was_contributory()`), added in review. **How to apply:** for crypto deliveries, walk a fixed checklist (nonce sourcing, contributory ECDH, domain separation, zeroization on every early-return path, error-oracle collapse) rather than relying on the test suite the same author wrote.

- **2026-07-04 — Cross-artifact consistency review pays.** Orchestrator review of the spec drafts caught a wire/DDL mismatch (op wire field `target_ver` had no `ops.target_version` column, which would have broken canonical-byte reconstruction for hash-chain verification) and a plaintext-set overreach vs PRD §6.3. **How to apply:** after any subagent delivers multi-document or code+spec output, diff the artifacts against each other and against the PRD before committing — don't review documents only in isolation.
